## AI 診斷日誌：本作業如何處理常見錯誤

### 1. Folium 地圖上經緯度填反（lat/lon 順序）

Folium 的座標順序必須是 `[latitude, longitude]`，不能寫成 `[longitude, latitude]`。作業說明中也特別提醒這是常見陷阱；若順序寫反，點位通常會跑到錯誤位置，甚至直接跑出臺灣範圍之外。我的程式在建立雨量站圖層時，明確使用 `location=[row["lat"], row["lon"]]`；在建立避難所圖層時，則使用 `location=[row.geometry.y, row.geometry.x]`。其中 `Point.x` 代表經度、`Point.y` 代表緯度，因此這樣寫可以確保 Folium 收到的是正確的 `[lat, lon]` 順序，而不是把經緯度顛倒。:contentReference[oaicite:0]{index=0} 

實作上，我先把分析完成後的 GeoDataFrame 轉回 `EPSG:4326`，再交給 Folium 視覺化。這一步也很重要，因為 Folium 本身預設就是以 WGS84 經緯度繪圖；若直接拿投影座標系的 `x/y` 當成經緯度，也會造成地圖位置異常。換句話說，這個問題不是只靠「欄位名稱看起來像 lat/lon」就能避免，而是要同時確認 **CRS 正確** 與 **Folium 的座標順序正確**。

### 2. CWA API 回傳 `-998` 導致地圖顏色異常

CWA 資料中，`-998` 並不是實際降雨量，而是「無資料」的缺值代碼。作業規範也明確要求在建立 GeoDataFrame 之前就先把 `-998` 過濾掉；如果不先處理，後續的 CircleMarker 顏色分級與 HeatMap 權重就會把 `-998` 當成真實數值，導致顏色異常、符號大小失真，甚至讓整體統計結果被極端負值污染。

我的做法是在 parser 階段先清理雨量欄位，把 `-998` 視為缺值代碼，而不是直接進入地圖樣式函式。這樣做的好處是，後面不論是 `rain_color()`、`rain_radius()`，還是 HeatMap 使用的 `rain_1hr` 權重，都只會吃到已經清洗過的合理數值。換句話說，我把錯誤值的處理放在資料前處理層，而不是等到視覺化時才修補，這樣整個分析鏈比較穩定，也比較不容易在不同 cell 中重複出錯。

### 3. `sjoin` 結果為空（CRS 未對齊）

`gpd.sjoin()` 最常見的錯誤之一，就是左右兩個 GeoDataFrame 使用了不同的 CRS。這份作業中，雨量站原始資料是 `EPSG:4326`，但 buffer 與距離分析必須使用公尺單位的投影座標系，因此作業要求先把雨量站轉為 `EPSG:3826`，再與 Week 3–4 的 shelters 做空間疊合；說明檔也直接提醒，若 `sjoin` 前 CRS 未對齊，結果很可能會是空的。:contentReference[oaicite:5]{index=5}

我在程式中先將分析用的 GeoDataFrame 統一到同一個投影座標系，再進行 buffer 與 `sjoin`。這樣做有兩個目的：第一，`buffer(5000)` 才真的代表 5000 公尺，而不是經緯度下毫無意義的 5000 度；第二，`sjoin` 會在相同的空間參考系中判斷 shelter 是否位於高雨量影響範圍內。也就是說，我把 CRS 對齊視為空間分析前的必要檢查，而不是當結果為空時才回頭排查。這個設計直接避免了「明明有豪雨站，卻一個 shelter 都配對不到」的問題。:contentReference[oaicite:6]{index=6}

### 4. HeatMap 在山區有盲區（測站分佈不均）

HeatMap 的本質是依靠測站點位與其權重來估計空間上的雨量強弱，因此它很容易受到「測站分布是否均勻」的影響。我的程式在建立 HeatMap 時，先從 `gdf_rainfall_map` 篩出 `rain_1hr > 0` 的站點，再把 `rain_1hr` 依最大值標準化成 `heat_weight`，最後用 `HeatMap(data=heat_data, ...)` 把點位與權重畫上地圖。這可以提升視覺穩定性，但無法根本消除高山區域因測站較少而產生的盲區。

換句話說，HeatMap 顯示的是「**有觀測站支撐的降雨強度分布**」，而不是完整、連續、無缺漏的真實降雨面。當山區測站密度低、谷地與稜線之間距離大時，熱力圖就可能對局部強降雨反應不足，造成視覺上像是「那一帶沒雨」的錯覺。這不是 Folium 套件本身錯誤，而是觀測資料空間分布不均造成的限制。我的程式透過只保留有效降雨站點、做權重標準化、並同時保留 CircleMarker 圖層來減少誤判：也就是說，HeatMap 用來看整體趨勢，CircleMarker 則用來保留單站真實數值，兩者互補使用。

此外，這份作業的歷史快照資料本身是由 CoLife 的真實觀測紀錄篩選並轉成 JSON，站點數雖然很多，但本質上仍受原始自動雨量站空間分布限制，因此山區盲區是一個資料層面的限制，而不是單靠調整 HeatMap 參數就能完全消除。這也是為什麼在災害應用上，我不只依賴 HeatMap，而是同時結合高雨量站 buffer、`sjoin`、地形風險與動態風險分級，讓判讀不會只建立在一張熱力圖上。
